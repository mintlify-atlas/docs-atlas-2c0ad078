## Overview

`ms-auth` is the authentication microservice for GEMS LMS. It is responsible for user registration, login, and account lifecycle management. All user identity operations are owned by this service. It uses PostgreSQL as its data store and communicates with the database via R2DBC, Spring's reactive, non-blocking database driver — keeping the service fully asynchronous from HTTP request to database query.

## Main class

```java
package com.gems.auth;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class AuthApplication {
  public static void main(String[] args) {
    SpringApplication.run(AuthApplication.class, args);
  }
}
```

`@SpringBootApplication` enables Spring Boot auto-configuration, component scanning across the `com.gems.auth` package, and starts the embedded Netty server.

## Configuration

```properties
# Server
server.port=${AUTH_PORT}
spring.application.name=ms-auth

# R2DBC (reactive PostgreSQL)
spring.r2dbc.url=${AUTH_R2DBC_URL}
spring.r2dbc.username=${AUTH_R2DBC_USERNAME}
spring.r2dbc.password=${AUTH_R2DBC_PASSWORD}

# Logging
logging.level.com.gems.auth=INFO
logging.level.org.springframework.web=INFO
logging.level.org.springframework.r2dbc=DEBUG

# Actuator
management.endpoints.web.exposure.include=health,info,metrics
management.endpoint.health.show-details=when-authorized
```

### Environment variables

| Variable | Description | Example |
|---|---|---|
| `AUTH_PORT` | Port the service listens on | `8082` |
| `AUTH_R2DBC_URL` | R2DBC connection URL for PostgreSQL | `r2dbc:postgresql://localhost:5432/authdb` |
| `AUTH_R2DBC_USERNAME` | Database username | `auth_user` |
| `AUTH_R2DBC_PASSWORD` | Database password | *(secret)* |

<Warning>
Never commit database credentials to source control. Supply `AUTH_R2DBC_PASSWORD` and other secrets via environment variables, a secrets manager, or Docker Compose environment configuration.
</Warning>

## Package structure

The service follows hexagonal architecture. Domain logic is isolated from both the HTTP layer and the database layer through explicit port and adapter boundaries.

```
com.gems.auth/
├── AuthApplication.java             # Application entry point
├── domain/
│   ├── entities/                    # Core domain objects (e.g. User)
│   ├── exceptions/                  # Domain-specific exceptions
│   └── values/                      # Value objects (e.g. Email, Password)
├── application/
│   ├── command/                     # Command objects (write operations)
│   └── gateway/                     # Outbound port interfaces
└── infrastructure/
    ├── driving/                     # Inbound adapters (HTTP controllers, WebFlux routes)
    └── driven/                      # Outbound adapters (R2DBC repositories, external clients)
```

## Database schema

The `users` table is the primary entity managed by `ms-auth`. It is initialized via `schema.sql` on startup.

```sql
CREATE TABLE IF NOT EXISTS users (
    user_id    BIGSERIAL PRIMARY KEY,
    username   VARCHAR(50)  UNIQUE NOT NULL,
    email      VARCHAR(100) UNIQUE NOT NULL,
    password   VARCHAR(255) NOT NULL,
    first_name VARCHAR(50)  NOT NULL,
    last_name  VARCHAR(50)  NOT NULL,
    is_active  BOOLEAN      NOT NULL DEFAULT true,
    created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);
```

### Column reference

| Column | Type | Constraints | Description |
|---|---|---|---|
| `user_id` | `BIGSERIAL` | Primary key | Auto-incrementing user identifier |
| `username` | `VARCHAR(50)` | Unique, not null | Login username |
| `email` | `VARCHAR(100)` | Unique, not null | Email address |
| `password` | `VARCHAR(255)` | Not null | Hashed password |
| `first_name` | `VARCHAR(50)` | Not null | Given name |
| `last_name` | `VARCHAR(50)` | Not null | Family name |
| `is_active` | `BOOLEAN` | Default `true` | Account active flag |
| `created_at` | `TIMESTAMP` | Default now | Record creation timestamp |
| `updated_at` | `TIMESTAMP` | Default now | Last update timestamp |

## Docker build

Uses the same two-stage pattern as other GEMS services:

1. **Build stage** — `eclipse-temurin:24-jdk`: builds the JAR via `./gradlew :ms-auth:build -x test`.
2. **Runtime stage** — `eclipse-temurin:24-jre`: runs the assembled JAR on port `8082`.

```dockerfile
FROM eclipse-temurin:24-jdk as builder
WORKDIR /app
COPY gradlew .
COPY gradle gradle
COPY build.gradle .
COPY settings.gradle .
COPY shared shared
COPY ms-auth ms-auth
RUN chmod +x gradlew
RUN ./gradlew :ms-auth:build -x test

FROM eclipse-temurin:24-jre
WORKDIR /app
COPY --from=builder /app/ms-auth/build/libs/*.jar app.jar
EXPOSE 8082
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \\
  CMD curl -f http://localhost:8082/actuator/health || exit 1
ENTRYPOINT ["java", "-jar", "app.jar"]
```

## Running locally

```bash
./gradlew :ms-auth:bootRun
```

Ensure the R2DBC environment variables are set and a PostgreSQL instance is reachable at the configured URL before starting.

<Tip>
Use Docker Compose to start `ms-auth` alongside its PostgreSQL database and the API Gateway in a single command. This removes the need to manage individual database connections during local development.
</Tip>
