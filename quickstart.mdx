---
title: Quickstart
description: Get the GEMS LMS API platform running locally in minutes using Docker Compose.
---

This guide walks you through cloning the repository, configuring environment variables, and starting all four microservices with a single command.

<Steps>
  <Step title="Check prerequisites">
    Ensure the following are installed before you begin:

    | Tool | Minimum version | Purpose |
    | --- | --- | --- |
    | Java (JDK) | 24 | Required to build and run services locally without Docker |
    | Docker | 24.x | Container runtime |
    | Docker Compose | v2 (bundled with Docker Desktop) | Orchestrates all services and databases |

    <Note>
      Java 24 is only required if you plan to run services directly with `./gradlew bootRun`. For the Docker Compose path, Docker is sufficient — the multi-stage Dockerfiles build the JARs inside the container using `eclipse-temurin:24-jdk`.
    </Note>

    Verify your setup:

    ```bash
    java -version
    docker --version
    docker compose version
    ```
  </Step>

  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/GEMS-INNOVATIONS/gems-lms-api.git
    cd gems-lms-api
    ```

    The monorepo contains five Gradle subprojects:

    ```
    gems-lms-api/
    ├── api-gateway/      # Routes all inbound traffic
    ├── ms-auth/          # Authentication and user management
    ├── ms-education/     # Course and content management
    ├── ms-admin/         # Admin user and platform management
    └── shared/           # Shared domain types and utilities
    ```
  </Step>

  <Step title="Configure environment variables">
    All services read configuration from environment variables. Create a `.env` file in the project root — Docker Compose and the PowerShell helper scripts both load it automatically.

    ```bash .env
    # API Gateway
    API_GATEWAY_PORT=8080

    # Education microservice
    EDUCATION_PORT=8081
    EDUCATION_DB_NAME=education_db
    EDUCATION_DB_USER=education_user
    EDUCATION_DB_PASSWORD=education_password
    EDUCATION_DB_PORT=5434

    # Auth microservice
    AUTH_PORT=8082
    AUTH_DB_NAME=auth_db
    AUTH_DB_USER=auth_user
    AUTH_DB_PASSWORD=auth_password
    AUTH_DB_PORT=5433

    # Admin microservice
    ADMIN_PORT=8083
    ADMIN_DB_NAME=admin_db
    ADMIN_DB_USER=admin_user
    ADMIN_DB_PASSWORD=admin_password
    ADMIN_DB_PORT=5432
    ```

    <Warning>
      Never commit the `.env` file to version control. It is already listed in `.gitignore`. Rotate all passwords before deploying to any non-local environment.
    </Warning>

    Each service constructs its R2DBC connection string at startup from the variables above. For example, `ms-auth` builds:

    ```
    r2dbc:postgresql://postgres-auth:${AUTH_DB_PORT}/${AUTH_DB_NAME}
    ```

    The hostnames (`postgres-auth`, `postgres-education`, `postgres-admin`) are internal Docker network aliases — they are not accessible from outside the `gems-lms-network` bridge network.
  </Step>

  <Step title="Start all services with Docker Compose">
    Build images and start all services in detached mode:

    ```bash
    docker compose up -d
    ```

    On first run, Docker builds all four service images using multi-stage builds (`eclipse-temurin:24-jdk` for the build stage, `eclipse-temurin:24-jre` for the runtime stage). Subsequent starts reuse the cached images and are much faster.

    <CodeGroup>

    ```bash Start (detached)
    docker compose up -d
    ```

    ```bash Start with rebuild
    docker compose up -d --build
    ```

    ```bash Stop all services
    docker compose down
    ```

    ```bash Stop and remove volumes
    docker compose down -v
    ```

    </CodeGroup>

    Docker Compose respects startup order: each microservice waits for its database container to pass the `pg_isready` healthcheck before starting. The `api-gateway` starts last, after all three microservices are running.

    <Tip>
      On Windows, you can also use the included PowerShell scripts to run services directly with Gradle (no Docker required): `./start-all-microservices.ps1` starts each service in a separate terminal window. `./stop-all-microservices.ps1` kills all processes on the configured ports.
    </Tip>
  </Step>

  <Step title="Verify all services are healthy">
    Each service exposes a Spring Boot Actuator health endpoint. Poll them to confirm a successful startup:

    <CodeGroup>

    ```bash API gateway (port 8080)
    curl http://localhost:8080/actuator/health
    ```

    ```bash Education service (port 8081)
    curl http://localhost:8081/actuator/health
    ```

    ```bash Auth service (port 8082)
    curl http://localhost:8082/actuator/health
    ```

    ```bash Admin service (port 8083)
    curl http://localhost:8083/actuator/health
    ```

    </CodeGroup>

    A healthy service responds with:

    ```json
    {"status":"UP"}
    ```

    You can also inspect container health status from Docker:

    ```bash
    docker compose ps
    ```

    All four service containers (`gems-api-gateway`, `gems-ms-education`, `gems-ms-auth`, `gems-ms-admin`) should show `healthy` in the `STATUS` column. Containers restart automatically (`restart: unless-stopped`) if they crash.

    <Note>
      Services have a 40-second `start_period` before Docker begins health checks. If containers appear unhealthy immediately after start, wait a moment and check again with `docker compose ps`.
    </Note>
  </Step>
</Steps>
